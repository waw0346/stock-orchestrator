## 🎯 검토 개요
현재 KOSPI200 선물과 현물 간의 스프레드(베이시스) 및 프로그램 매매 동향을 실시간으로 추적하여 매수 타이밍을 정밀 제어하기 위해 제안된 **[1] WebSocket Push Client 도입**과 **[2] 인메모리 데이터 저장소(Redis 등) 전환**의 타당성을 검토합니다.

## ⚖️ 제안 1: 실시간 웹소켓 푸시(WebSocket Push) 도입 검토

### 1. 반영 가능/불가능 여부 검토
> [!IMPORTANT]
> **결론: 반영 가능 (조건부 하이브리드 구현)**

*   **현재 아키텍처 상태:** 현재 프로젝트의 [kiwoom_rest_client.py](file:///c:/Users/kjw03/Desktop/stock%20orchestrator/scripts/kiwoom_rest_client.py)는 Stateless HTTP POST 통신 기반으로 설계되어 있습니다.
*   **기술적 제약 사항:** 키움 OpenAPI+는 기본적으로 Windows COM/OCX(ActiveX) 환경을 기반으로 작동하므로, 순수 파이썬 스크립트에서 다이렉트 웹소켓 연결을 지원하지 않는 경우가 있습니다.
*   **반영 솔루션:** 
    1.  키움증권에서 공식 제공하는 **REST/WebSocket Gateway API**를 사용하는 경우, 파이썬의 `websockets` 라이브러리를 통해 즉시 구현 가능합니다.
    2.  ActiveX 버전만 제공될 경우, 백그라운드에서 `PyQt5` 기반의 실시간 시세 수집 데몬(Daemon)을 구동하고, 이 데몬이 수집한 시세를 로컬 웹소켓 서버를 통해 메인 엔진에 푸시해 주는 **로컬 중계 데몬(Hybrid Broker)**을 제작하여 100% 극복 가능합니다.

### 2. 전환 시 실제 운영 및 구현 가능성 검토
*   **실행 모델의 변화:** 기존의 배치성 스크립트 실행 모델(`.ps1`을 실행하고 결과 저장 후 종료)에서 **상시 실행형 데몬(Daemon) 프로세스**로 전환해야 합니다.
*   **운영 오버헤드:** 장 시작(09:00 KST) 시 자동으로 웹소켓 세션을 수립하고, 장 마감(15:30 KST) 시 세션을 안전하게 정리하며, 네트워크 단절 시 **자동 재연결(Auto-Reconnect) 및 하트비트(Heartbeat)**를 처리하는 예외 처리 코드가 추가되어야 합니다. 이는 개발 공수가 필요하지만 상시 가동형 시스템으로서 충분히 운영 가능합니다.

### 3. 매수 방법에 있어 효율적인 방법인가?
> [!TIP]
> **체결 효율 극대화 (실시간 대응을 위해 필수적)**

*   **REST Polling (기존):** 10~30초 단위로 시세를 가져오는 동안 스프레드 괴리에 의한 프로그램 투매는 이미 완료되어 하락의 끝자락이나 반등 직전에 매수를 시도하는 **뒷북 매수(Slippage)**가 발생합니다.
*   **WebSocket Push (전환):** 지연율이 **수 밀리초(ms)** 단위로 압축되어, 지수 급락 시 외국인의 선물 매도로 인한 베이시스 급락을 즉시 포착합니다. 이에 따라 프로그램 매도 폭탄이 현물 호가에 반영되기 **직전에 매수를 HOLD**하고, 투매가 멈추고 스프레드가 복귀하는 **최적의 바닥 가격에서 매수를 체결**시키는 극도로 효율적인 진입이 가능해집니다.

## ⚖️ 제안 2: 인메모리 데이터 저장소(Redis 등) 전환 검토

### 1. 반영 가능/불가능 여부 검토
> [!IMPORTANT]
> **결론: 즉시 반영 가능 (매우 용이함)**

*   **현재 아키텍처 상태:** 모든 스크립트가 `picks/cache/` 디렉토리 내의 JSON 파일 읽기/쓰기에 의존하고 있습니다.
*   **기술적 제약 사항:** 로컬 개발 환경(Windows)에 Redis 서버를 추가 설치하는 것은 인프라 종속성을 높일 수 있습니다.
*   **반영 솔루션 (대안):**
    *   **대안 A (권장):** 외부 설치 없이 파이썬 내장 라이브러리인 **메모리 맵 파일(`mmap`)** 또는 **메모리 내 SQLite DB (`sqlite3` with `:memory:`)**를 사용하여 파일 I/O 병목을 해결합니다.
    *   **대안 B:** 가벼운 Redis for Windows 서비스를 로컬에 띄워 백그라운드 데이터 캐시 레이어로 연결합니다. 

### 2. 전환 시 실제 운영 및 구현 가능성 검토
*   **데이터 휘발성 리스크:** 인메모리 DB는 프로세스가 꺼지면 장중 수집 데이터가 소실됩니다.
*   **운영 솔루션:** 이를 보완하기 위해 장중에는 모든 데이터 처리를 메모리(RAM) 상에서 초고속으로 수행하되, **장 마감 시(15:30) 또는 1시간 주기**로 메모리 내의 수급 데이터와 지표들을 백업용 JSON/CSV 파일로 아카이빙하는 **영속화 모듈(Persistency Module)**을 장착합니다. 기존의 파일 쓰기 함수를 호출하는 방식이므로 구현 및 운영이 매우 간단합니다.

### 3. 매수 방법에 있어 효율적인 방법인가?
> [!TIP]
> **체결 효율 및 시스템 안정성 극대화**

*   **동시 파일 쓰기 충돌 방지:** 여러 수집 스크립트(외국인 수급, 시황 레이더, 모멘텀 스크리너 등)가 동시에 JSON 파일을 열고 쓰면서 발생하던 **파일 락(File Lock) 및 JSON 파일 깨짐(JSONDecodeError) 버그가 완벽하게 방지**됩니다.
*   **지연 속도 개선:** 데이터 쓰기/읽기 지연시간이 마이크로초($\mu s$) 단위로 단축되어, "이상 베이시스 신호 포착 즉시 매수 정지/실행" 알고리즘이 연산 지연 없이 즉각 실행창구로 전달되므로 매수 체결 효율이 극대화됩니다.

| **반영 가능성** | **가능 (하이브리드 중계 데몬 필요)** | **즉시 가능 (파이썬 내장 mmap/sqlite 활용)** |
