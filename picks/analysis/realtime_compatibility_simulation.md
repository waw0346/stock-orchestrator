### 📈 시장 상황 설정
*   **글로벌 환경:** 전일 미 증시 급락 여파로 장초반 갭하락 후 외국인 매도세 집중.
*   **선물-현물 베이시스 추이:** 
    *   09:00 (시가): -1.2 (정상 범위 하단)
    *   10:30 (급락기): -2.8 (괴리 극대화, 프로그램 매도 4천억 출회)
    *   13:00 (바닥 횡보): -2.5
    *   15:30 (마감): -2.3

### 🚨 구조적/호환성 문제 진단

#### 1. API 네트워크 랙 및 시간 왜곡 (Network Latency Gap)
*   **현상:** 증권사 서버로부터 실시간 푸시되는 `basis` 데이터 수집 시, 네트워크 일시 지연(약 5~15초)이 발생할 경우.
*   **문제점:** 베이시스가 -2.8을 찍으며 비차익 프로그램 매도가 쏟아지는 시점은 **10시 30분 00초**인데, 분석 엔진이 이를 캐시 파일에서 읽는 시점은 지연으로 인해 **10시 30분 15초**가 됨. 이미 현물 호가창에 수십 억의 매도 포지션이 체결되어 주가가 급락한 이후 뒤늦게 `basis_anomaly_alert`가 발생하여 **Capital Protection Gate의 매수 정지 명령이 뒷북으로 실행(Slippage Risk)**됨.

#### 2. 파일 I/O 충돌에 의한 덮어쓰기 유실 (JSON Write Override Collision)
*   **현상:** 리팩토링된 파이프라인(`run_us_close_korea_pipeline.ps1`)이 동일 프로세스 스레드 내에서 파이썬 스크립트들을 촘촘하게 호출하고, 동시에 `run_market_radar.py`가 초당 수회 단위로 `market_data_snapshot.json`을 읽고 씀.
*   **문제점:** 별도의 락(Lock) 메커니즘 없이 파일 입출력 속도만 가속될 경우, `collect_market_data.py`가 쓰기 작업을 완료하지 않은 시점(Partial Write 상태)에 `market_radar.py`가 읽기를 시도하여 **`JSONDecodeError` 또는 빈 객체를 가져오는 오류**를 발생시킴. 이는 실시간 모니터링 시스템의 불시 다운으로 이어질 수 있음.

## 2️⃣ [Simulation 2] 2026-06-09: V자형 급반등 장세 시뮬레이션

### 📈 시장 상황 설정
*   **글로벌 환경:** 장초반 패닉셀로 코스피 지수가 2% 넘게 폭락했으나, 연기금의 구원투수 매수세로 오후에 보합권까지 급반등.
*   **선물-현물 베이시스 추이:**
    *   09:15 (패닉기): -3.2 (연중 최저치)
    *   11:00 (소강기): -1.8
    *   13:30 (반등기): -0.4
    *   15:00 (안정기): +0.2

### 🚨 구조적/호환성 문제 진단

#### 1. 통계적 극단값에 따른 과잉 방어 (False Positive Alert Lock)
*   **현상:** 장초반 극단적인 베이시스 왜곡(-3.2)을 반영하면서 이동평균과 표준편차($\sigma$)의 밴드가 과도하게 하단으로 팽창함.
*   **문제점:** 오후 들어 시장이 급격하게 회복되고 베이시스가 -0.4 수준으로 정상 복구되었음에도 불구하고, 오전의 극단적인 수치 때문에 볼린저 밴드의 임계값이 비정상적으로 넓게 유지됨. 그 결과 시장이 안정을 되찾고 매수 진입이 가능한 최적의 골든타임임에도 시스템은 "비정상 베이시스 지속 상태"로 잘못 판단해 **매수 게이트 차단(HOLD)을 장 마감 때까지 풀지 않는 기회비용 유실 문제**가 발생함.

#### 2. 비대화형 파이프라인의 에러 전파 차단 실패 (Pipeline Chain Failure)
*   **현상:** `run_weekly_flow_momentum_update.ps1`에서 `-NoPrompt` 옵션으로 실행 시, 만약 네트워크 단절 등으로 주간 수급 데이터 수집이 비정상화되어 `throw`로 강제 종료될 때.
*   **문제점:** 서브 프로세스 분리 기동 방식(기존)에서는 하위 스크립트가 에러로 멈춰도 메인 파이프라인은 다음 단계로 넘어가거나 독자적으로 예외 처리를 수행할 수 있었으나, 닷소싱 및 직접 호출(개선안) 환경에서는 하나의 예외가 **전체 파이프라인 전체를 즉시 Crash(종료)**시켜 시황 감시 엔진 가동 전체가 중단되는 현상이 생김.

## 🛡️ 시뮬레이션 기반 보완 대책 (Mitigation & Action Items)

위 가상 시뮬레이션을 통해 발견된 4대 문제점을 해결하기 위해 다음과 같이 구현을 보완합니다.

1.  **지연 필터 및 호가 스프레드 하이브리드 감시:**
    *   단순 베이시스 수치뿐만 아니라, KOSPI200 주가지수선물의 체결량 가속도(Volume Acceleration) 및 호가 잔량 스프레드를 결합하여, 지연이 발생하더라도 가격 움직임의 성질을 보고 선제적으로 게이트를 잠그는 로직을 추가합니다.
2.  **안전 파일 라이팅 메커니즘 (Atomic Write) 도입:**
    *   JSON 파일 작성 시 직접 덮어쓰지 않고, 임시 파일(e.g., `.tmp`)에 먼저 쓴 뒤 성공적으로 완료되면 `os.replace()`를 호출해 원자적(Atomic)으로 덮어쓰도록 설계하여 파일 깨짐 및 동시성 충돌을 물리적으로 방지합니다.
3.  **지수 이동평균(EMA) 윈도우 단축화:**
    *   극단값에 의한 통계 왜곡을 줄이기 위해 단순 이동평균 대신 **감쇠 계수($\alpha$)가 높은 지수 이동평균(EMA)**을 활용하고, 장초반 변동성은 30분 이내에 가중치가 반감되도록 설계하여 복구 시 속히 정상 범주로 회귀하도록 보정합니다.
4.  **파이프라인 단계별 격리 에러 트래핑 (Isolated Try-Catch):**
     *   부모 파이프라인 내에서 각 파이썬 및 파워쉘 태스크를 독립적인 `try-catch` 블록으로 확실히 감싸고, 실패 시 전체 중단 대신 경고 알림(`pending.json`에 경고 발행) 후 안전하게 다음 시황 분석 단계를 이어나가도록 격리 처리를 철저히 하겠습니다.
